## 1. Une donnée, trois représentations

La même tâche existe sous trois formes, une par monde :

**Dans le navigateur (TypeScript)** — `frontend/src/types/task.ts` :
```typescript
interface Task {
  id: number;
  title: string;
  done: boolean;
  dueDate: string | null;
  createdAt: string;
}
```

**Dans le backend (Java)** — `backend/.../task/Task.java` :
```java
@Entity
public class Task {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String title;
    private boolean done;
    private LocalDate dueDate;
    private Instant createdAt;
}
```

**Dans la base (SQL)** — table générée par Hibernate :
```sql
create table task (
    id         bigint generated by default as identity primary key,
    title      varchar(200) not null,
    done       boolean not null,
    due_date   date,
    created_at timestamp(6) with time zone
);
```

Et entre le front et le back voyage une **quatrième forme, pivot** : le JSON.

```json
{ "id": 5, "title": "Réviser", "done": false,
  "dueDate": "2026-07-10", "createdAt": "2026-07-03T20:35:35Z" }
```

```
TypeScript  ◄──── JSON ────►  Java  ◄──── SQL ────►  table H2
(navigateur)   (le pivot)   (backend)              (fichier .mv.db)
```

## 2. Les deux traducteurs : Jackson et Hibernate

Personne n'écrit ces conversions à la main. Deux bibliothèques, embarquées
par nos starters Spring Boot, font chacune UNE frontière :

| Traducteur | Frontière | Déclenché par |
|---|---|---|
| **Jackson** | JSON ⇄ objets Java | `@RequestBody` (JSON→Java) et le `return` d'un `@RestController` (Java→JSON) |
| **Hibernate** | objets Java ⇄ lignes SQL | `repository.save()`, `findAll()`, `findById()`... |

Côté navigateur, la traduction JSON ⇄ objets JavaScript est native :
`response.json()` (lecture) et `JSON.stringify(...)` (écriture) — visibles
dans notre `api/tasks.ts`.

**Le point clé : le JSON est le seul contrat entre le front et le back.**
Le front ne sait pas que le back est en Java ; le back ne sait pas que le
front est en React. Chacun ne connaît que le JSON qui circule. (C'est
pour ça qu'on pourrait remplacer React par une app mobile sans toucher
au backend.)

## 3. La correspondance des noms

Comment `dueDate` devient `due_date` puis redevient `dueDate` ? Chaque
traducteur applique sa convention :

| TypeScript (front) | JSON (pivot) | Java (back) | SQL (BDD) |
|---|---|---|---|
| `task.dueDate` | `"dueDate"` | `private LocalDate dueDate` | `due_date` |

- **Jackson** génère les clés JSON à partir des **getters/setters** Java :
  `getDueDate()` → clé `"dueDate"` (il retire `get` et met la minuscule).
  Dans l'autre sens, la clé `"dueDate"` du JSON entrant est injectée via
  `setDueDate(...)`.
- **Hibernate** convertit le camelCase Java en **snake_case** SQL :
  `dueDate` → colonne `due_date` (stratégie de nommage par défaut de
  Spring Boot).
- **TypeScript** n'invente rien : notre interface `Task` doit être écrite
  avec les MÊMES noms que les clés JSON. C'est un engagement qu'on prend,
  pas une vérification automatique (voir §7).

## 4. La correspondance des types

| TypeScript | JSON | Java | SQL (H2) |
|---|---|---|---|
| `number` | `5` | `Long` | `bigint` |
| `string` | `"Réviser"` | `String` | `varchar(200)` |
| `boolean` | `true` | `boolean` | `boolean` |
| `string \| null` | `"2026-07-10"` ou `null` | `LocalDate` | `date` |
| `string` | `"2026-07-03T20:35:35Z"` | `Instant` | `timestamp with time zone` |

Deux remarques :

- **JSON n'a pas de type date.** Les dates voyagent en texte (format ISO
  8601 : `"2026-07-10"`). Jackson sait convertir ce texte vers/depuis
  `LocalDate`/`Instant` ; côté front, on garde des `string` et on ne
  convertit en `Date` JavaScript que pour calculer.
- **`null` et les optionnels.** Côté SQL, une colonne sans `not null`
  accepte `NULL` ; côté Java, seuls les types objets (`Long`, `LocalDate`)
  peuvent être null (pas les primitifs comme `boolean`) ; côté TypeScript,
  on l'écrit explicitement : `string | null`.

## 5. La correspondance des URLs

Le lien entre un appel `fetch` du front et une méthode Java se fait par
l'URL + le verbe HTTP, déclarés de chaque côté :

**Front** (`api/tasks.ts`) :
```typescript
fetch(`http://localhost:8080/api/tasks/${id}`, { method: 'PATCH', ... })
```

**Back** (`TaskController.java`) :
```java
@RequestMapping("/api/tasks")          // préfixe de classe
...
@PatchMapping("/{id}")                 // PATCH /api/tasks/{id}
public Task update(@PathVariable Long id, ...)
```

Correspondances : `PATCH` ⇄ `@PatchMapping`, le `${id}` de l'URL front ⇄
le `{id}` du mapping ⇄ le paramètre `@PathVariable Long id`.

## 6. Le voyage complet : cocher une tâche, tracé de bout en bout

L'utilisateur clique la case de la tâche 2. Voici TOUT ce qui se passe,
avec le fichier responsable à chaque étape :

**Aller (navigateur → BDD)**

1. `TaskItem.tsx` — le clic déclenche `onChange` → appelle `onToggle(task)`
   reçu en props (les événements remontent au parent).
2. `TasksPage.tsx` — `toggleMutation.mutate(task)` s'exécute → appelle
   `updateTask(2, { done: true })`.
3. `api/tasks.ts` — `fetch` construit la requête HTTP :
   `PATCH /api/tasks/2`, corps `{"done":true}` (via `JSON.stringify`).
4. **CORS** — le navigateur vérifie que `localhost:8080` autorise l'origine
   `localhost:5173` (notre `CorsConfig.java`), puis laisse partir la requête.
5. **Tomcat** (port 8080) reçoit le texte HTTP brut et le décode.
6. **Spring** consulte sa table de routage : `PATCH /api/tasks/{id}` →
   `TaskController.update(id=2, changes)`.
7. **Jackson** transforme le corps `{"done":true}` en
   `UpdateTaskRequest(title=null, done=true, dueDate=null)`.
8. `TaskController.java` — `findById(2)` charge la tâche, `setDone(true)`,
   puis `repository.save(task)`.
9. **Hibernate** traduit en SQL : `update task set done=? where id=?` et
   l'exécute sur le fichier H2. La donnée est persistée.

**Retour (BDD → navigateur)**

10. Le controller retourne l'objet `Task` mis à jour → **Jackson** le
    sérialise en JSON `{"id":2, "done":true, ...}` → réponse `200`.
11. `api/tasks.ts` — `response.json()` → objet TypeScript.
12. `TasksPage.tsx` — `onSuccess` → `invalidateQueries(['tasks'])` :
    TanStack Query relance `fetchTasks` (GET), reçoit la liste fraîche.
13. **React** re-rend les composants dont les données ont changé : la case
    s'affiche cochée, le titre passe en barré. Aucun rechargement de page.

Treize étapes, et tu n'as écrit que les fichiers des étapes 1-3, 8 et 12 —
le reste est fourni par Tomcat, Spring, Jackson, Hibernate et TanStack.

## 7. Qui garantit la cohérence ? (spoiler : toi)

Point important : **rien ne vérifie automatiquement** que l'interface
TypeScript, l'entité Java et la table sont alignées.

- Java ⇄ SQL : cohérence **garantie** (Hibernate génère la table depuis
  l'entité — une seule source de vérité).
- TypeScript ⇄ JSON ⇄ Java : contrat **implicite**. Si tu renommes
  `title` en `name` dans `Task.java` sans toucher au front, rien ne casse
  à la compilation : le front enverra `"title"`, Jackson ne trouvera pas
  de `setTitle`, le champ restera null... bug silencieux.

Règle pratique pour StudyMate : **toute modification d'une entité côté
Java doit être répercutée dans l'interface TypeScript correspondante**
(`types/*.ts`), et inversement. Les projets plus gros automatisent ça
(génération des types depuis une spec OpenAPI) — on pourra le faire plus
tard si le besoin s'en fait sentir.
